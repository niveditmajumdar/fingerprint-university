Validation rules & caching policy
Suspect score
⚠️
Step-up threshold ≥ 12
When a device's suspect score reaches 12 or above, a verification modal is shown before allowing certification submission. The score is a weighted sum of active signals.
🔢
Signal weights
Bot detected +6 VPN active +4 Incognito mode +3 Low confidence (<50%) +4
VPN detection
🌐
Login page Active
After sign-in, the Server API is called to get authoritative VPN status. If VPN is detected, a persistent modal appears requiring the user to dismiss before proceeding. Uses offset-based timezone comparison to avoid false positives from IANA aliases (e.g. Asia/Kolkata vs Asia/Calcutta).
🏆
Certification tab Active
VPN detection is re-checked when the certification tab is opened (subject to the 2-minute TTL). If VPN is active, the suspect modal is triggered before the quiz renders. At submission, VPN status also blocks submission and triggers the verify-code modal.
⭐
Review submission Active
VPN detected at review submission time causes the review to be stored as status = flagged rather than approved. Gift card is withheld pending manual review.
Incognito mode
🕵️
Detection source Server API only
Incognito status is NOT available in the JS agent client response — it is a Smart Signal available only via the Server API. The JS agent cannot detect incognito mode itself; the server-side call after login provides the authoritative value.
📊
Effect on suspect score
Incognito mode adds +3 to the computed suspect score when detected. Combined with VPN (+4), this reaches 7 — still below the threshold of 12. Bot detection (+6) combined with incognito (+3) would push to 9. Bot + VPN hits 10.
Timezone mismatch
🕐
IP vs browser timezone Active
The IP geolocation timezone (from Server API ipInfo.v4.geolocation.timezone) is compared against the browser's reported timezone using UTC offset arithmetic — not string comparison. IANA aliases (e.g. Asia/Kolkata = Asia/Calcutta = UTC+5:30) are handled correctly.
Caching & API call policy
⏱️
Certify tab TTL 2 minutes
Clicking "Get certified" only calls the Server API if the last call was more than 2 minutes ago. Within the TTL window, cached signals from sessionStorage are used and validations still run. The TTL timestamp (_lastServerApiCallAt) is shared across all Server API calls — footer Smart Signals, review tab, and login all count.
📋
Review tab TTL 2 minutes
Same 2-minute window. Additionally, if the certification test was submitted within 2 minutes of opening the review tab, the API call is skipped entirely — the cert submission signals are considered fresh enough.
🔴
Review submission Always fresh
The Server API call at review submission is mandatory — no TTL applies. This is because the review carries real monetary value ($20 gift card), and the fresh visitorId is compared against the session to detect session transfer attacks.
💾
Cache layers
L1 sessionStorage Per-tab, cleared on tab close. Incognito switch clears it.
L2 Express session Server-side, persists across page navigation.
L3 PostgreSQL devices Persistent across restarts. last_api_call_at column enforces server-side TTL.
✅
Cert submission device check JS agent only
At certification submission, a fresh JS agent call gets a new visitorId which is compared against the session value. No Server API call is made — cached suspect score from the tab-open call is used. This detects session cookie transfer to a different device.
Review deduplication
🔒
Check 1: Visitor ID Device-level
The visitor_id column in reviews has a UNIQUE constraint. Same device → blocked regardless of email used.
📧
Check 2: Email Identity-level
Email is checked case-insensitively across all reviews. Same email on a different device → blocked. Prevents gifting to the same person across multiple devices.
🔀
Check 3: Session mismatch Anti-hijack
The fresh visitorId at submission must match the session's visitorId. Detects session cookie copied to another device.
